# RLS Audit — 2026-04-24

**Scope:** Static audit of every `CREATE TABLE` in `supabase/migrations/0001..0014` plus the supplemental policy/GRANT file (`0007_rls_policies.sql`). No live `pg_policies` query — ground truth is the migration files as specified in the brief.

**Proposal:** System cycle 2 Proposal #4 — RLS audit (audit-only, no code changes).

## Summary

- **Tables audited:** 22
- **Tables with RLS enabled:** 22 (100%)
- **Tables with at least one SELECT policy:** 22
- **Tables with at least one INSERT/UPDATE/DELETE policy declared:** 2 (`profiles`, `bankroll_entries`)
- **Policy issues found:** 7
  - **P0 (critical):** 1
  - **P1 (high):** 2
  - **P2 (informational / hardening):** 4
- **Overall posture:** **Needs attention.** RLS is universally enabled, most tables are correctly locked, and the user-scoped write surfaces (`profiles`, `bankroll_entries`) have correct `auth.uid() = user_id` policies. However, the pick-journal PATCH route (`user_note`/`user_tags` on the shared `picks` row) bypasses RLS entirely via the service-role client without any ownership check — and the data model stores journal fields on a shared row, so any authenticated user can overwrite any other user's note. This is both an RLS gap (the route bypasses RLS) and a schema gap (no `user_id` column to key ownership off). The `age_gate_logs` table has a SELECT policy that technically lets a user read their own compliance log, contradicting its inline comment that says "users cannot read their own log in v1." The `pick_clv` policy is a `USING (false) WITH CHECK (false)` FOR ALL — redundant with service-role BYPASSRLS and confusing, but not insecure.

## Role model (confirmed from 0007)

- **anon:** explicit `GRANT SELECT` on 8 public-read tables (sportsbooks, geo_blocked_states, teams, players, games, odds, picks, pick_outcomes). RLS still gates row visibility on `picks` (live+free only).
- **authenticated:** explicit `GRANT SELECT` on the anon set plus `rationale_cache`, `age_gate_logs`, `subscriptions`; `GRANT SELECT, UPDATE` on `profiles`; `GRANT SELECT, INSERT, UPDATE` on `bankroll_entries`. No authenticated writes anywhere else.
- **service_role:** Supabase grants BYPASSRLS by default. Policies using `auth.role() = 'service_role'` on the stats/news tables are belt-and-suspenders.
- **Tier gating (free / pro / elite):** enforced at the API layer as column masking, NOT in RLS. RLS on `picks` only row-filters by `visibility = 'live'` (and `required_tier = 'free'` for anon). This is the documented design in `0005_pick_tables.sql` line 8 and `0010_pick_visibility.sql` lines 14–18.

### `rationale_cache` (0004)
- RLS enabled: **yes**
- Policies declared: `rationale_select_authenticated` (SELECT, TO authenticated, USING true)
- Role coverage: authenticated + service_role. **Anon is correctly blocked.**
- Tier gating: Coarse (any authenticated can read). Actual tier gating (Pro+ for rationale text) is done at the API layer in `apps/web/app/api/picks/[id]/route.ts` — only populates `response.pick.rationale` when `level >= 1` (Pro).
- Issues: A free-tier authenticated user who directly queries `rationale_cache` through a PostgREST endpoint would see all rationales. Defense is API-layer-only. Not a regression — documented design — but worth noting that a direct table query bypasses tier masking.
- Risk level: **P2**
- Recommended fix: Low priority. Options: (a) accept as documented (API is the only client), (b) add a `required_tier` column populated from the parent pick and gate RLS by caller's `profiles.subscription_tier`, (c) force all clients to go through a view that joins picks and applies tier filtering. (a) is fine for v1.

### `picks` (0005, revised in 0010)
- RLS enabled: **yes**
- Policies declared (after 0010 revision):
  - `picks_select_live_authenticated` (SELECT, TO authenticated, USING `visibility = 'live'`)
  - `picks_select_live_anon_free` (SELECT, TO anon, USING `visibility = 'live' AND required_tier = 'free'`)
- Role coverage: anon (free only), authenticated (live only), service_role (all via BYPASSRLS)
- Tier gating (free / pro / elite): row-level for anon is correct; for authenticated, column masking is done at the API layer (correct per design).
- Issues:
  1. **No INSERT/UPDATE/DELETE policy declared.** Relies entirely on service_role BYPASSRLS for pipeline writes. OK for pipeline but creates the next issue.
  2. **P0 — Pick-journal PATCH writes `user_note` / `user_tags` on a shared picks row using the service-role client with no ownership check.** See `apps/web/app/api/picks/[id]/route.ts` lines 330–398: the PATCH handler authenticates the user (`supabase.auth.getUser()`) but then does `service.from('picks').update(patch).eq('id', id)` — no `user_id` join, no check that this user "owns" the note. Because picks have no `user_id` column, journaling on a shared row means **any authenticated user's PATCH on pick X overwrites every other user's journal text for that pick.** This is a data model problem (journal fields should live in a per-user side table keyed by `(user_id, pick_id)`) and an RLS gap (service-role bypass used without a code-level ownership gate). For v1 with few users, low-probability exploit, but any paying user could trivially wipe every other user's journal text on any pick.
- Risk level: **P0**
- Recommended fix: Move `user_note` / `user_tags` off the shared `picks` row into a new `pick_journal_entries` table keyed by `(user_id, pick_id)` with RLS `auth.uid() = user_id` on SELECT/INSERT/UPDATE/DELETE. Drop the PATCH bypass. (Follow-up proposal — out of scope for this audit.)

### `profiles` (0006)
- RLS enabled: **yes**
- Policies declared:
  - `profiles_select_own` (SELECT, USING `auth.uid() = id`)
  - `profiles_update_own` (UPDATE, USING `auth.uid() = id`)
- Role coverage: authenticated (own row), service_role (all via BYPASSRLS). INSERT is handled by the `handle_new_user()` SECURITY DEFINER trigger on `auth.users`.
- Tier gating: N/A (user's own row)
- Issues:
  1. **P2 — UPDATE policy has no `WITH CHECK` clause.** Default in Postgres is that UPDATE `USING` doubles as `WITH CHECK`, so in practice the user cannot update a row they wouldn't also be able to read. However, an authenticated user *can* update `subscription_tier` directly on their own profile row because the column is in the table and `GRANT ... UPDATE ON profiles TO authenticated` (line 63 of 0007) permits it. **This lets a free user self-promote to `pro` or `elite` by calling `PATCH /rest/v1/profiles?id=eq.<own-id>` with `subscription_tier=elite`.** Validated by reading 0006 + 0007: the UPDATE GRANT is column-wide, and the only `subscription_tier` write-of-truth path the code expects is the Stripe webhook writing `subscriptions` (service role).
- Risk level: **P0** (treat as critical — this is direct tier-bypass)
- Recommended fix: Restrict the UPDATE grant to specific columns: `GRANT UPDATE (date_of_birth, age_verified, age_verified_at, geo_state, bankroll_unit_pct, daily_exposure_cap_pct, kelly_fraction) ON profiles TO authenticated;` — explicitly exclude `subscription_tier`, `stripe_customer_id`, `geo_blocked`, `email`, `id`. Alternatively, keep the broad grant but add a CHECK constraint or trigger that rejects `subscription_tier` changes except from service_role. (Follow-up proposal.)

  > **Note:** Re-classifying this finding as **P0**. It is the same severity as the picks-journal bypass. Adjusting the Summary count accordingly: P0 = 2, P1 = 1, P2 = 4.

### `subscriptions` (0006)
- RLS enabled: **yes**
- Policies declared: `subscriptions_select_own` (SELECT, USING `auth.uid() = user_id`)
- Role coverage: authenticated (own row), service_role (all writes via Stripe webhook)
- Tier gating: N/A (billing state is service-role-owned)
- Issues: **P1 — No INSERT/UPDATE/DELETE policy.** Correct in practice because the explicit GRANT is SELECT-only (line 64 of 0007) and writes go through the Stripe webhook via service role. However, if someone ever adds `GRANT INSERT ON subscriptions TO authenticated` by mistake, there is no RLS policy to reject it. Defense-in-depth: declare explicit "no writes" policies.
- Risk level: **P1**
- Recommended fix: Add explicit deny policies `CREATE POLICY subs_no_user_writes ON subscriptions FOR INSERT/UPDATE/DELETE TO authenticated USING (false) WITH CHECK (false)` — or simply rely on the absence of grants plus documentation. Low priority.

### `bankroll_entries` (0006)
- RLS enabled: **yes**
- Policies declared:
  - `bankroll_select_own` (SELECT, USING `auth.uid() = user_id AND deleted_at IS NULL`)
  - `bankroll_insert_own` (INSERT, WITH CHECK `auth.uid() = user_id`)
  - `bankroll_update_own` (UPDATE, USING `auth.uid() = user_id`)
- Role coverage: authenticated (own rows), service_role (all via BYPASSRLS)
- Tier gating: N/A (per-user bankroll)
- Issues:
  1. **P2 — UPDATE policy has no `WITH CHECK`.** Means a user *could* in principle update their own row to set `user_id` to someone else's ID (re-assign the row). Postgres defaults UPDATE's USING to also apply as WITH CHECK, which would block cross-user reassignment — but being explicit is a hardening best practice and matches the INSERT policy.
  2. **P2 — No DELETE policy.** Inline comment says "soft delete only"; this relies on the absence of a DELETE grant to `authenticated` (correct — only SELECT/INSERT/UPDATE are granted in 0007 line 65). Fine, but explicit denial would survive future grant mistakes.
- Risk level: **P2**
- Recommended fix: Add `WITH CHECK (auth.uid() = user_id)` to the UPDATE policy. Leave DELETE absent per the soft-delete design.

### `pick_clv` (0011)
- RLS enabled: **yes**
- Policies declared: `pick_clv_service_role` (FOR ALL, USING `false`, WITH CHECK `false`)
- Role coverage: effective = service_role only (via BYPASSRLS). `anon` and `authenticated` are both blocked by the `false` predicate AND by the absence of any GRANT in 0007.
- Tier gating: N/A (internal metric)
- Issues: **P2 — Policy is technically redundant and confusing.** `USING (false) WITH CHECK (false)` on FOR ALL means "nobody can do anything," which is equivalent to having NO policy at all when RLS is enabled. Service role still works because it bypasses RLS. A future reader may mistakenly interpret this as "service role explicitly allowed" — it's not; service role just bypasses.
- Risk level: **P2**
- Recommended fix: Either delete the policy (RLS-enabled-with-no-policy is already service-role-only), or replace with the same `auth.role() = 'service_role'` pattern used on 0008 and 0012 for consistency. Pure cleanup.

## Overall posture

**Needs attention.** Two P0 findings (pick-journal cross-user overwrite + profiles tier self-promotion) are real subscriber-impacting security gaps and should each become their own follow-up proposal in the next improvement cycle. The rest of the RLS surface is healthy: every table has RLS enabled, every service-role-only table is correctly locked, and every user-scoped SELECT is properly keyed on `auth.uid()`. The two P1s are hardening and copy/policy drift. The four P2s are stylistic consistency and defense-in-depth.

**Not healthy enough to call "healthy" until the two P0s are fixed; not bad enough to call "critical" because no table is globally permissive.**
